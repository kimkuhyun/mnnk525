# POLARIS DB 설계 — 00. 전체 개요

처음 보는 분을 위한 한 줄 요약 — **한국 공시(DART)를 근거로 답하는 GraphRAG 에이전트**의 데이터베이스 설계입니다.
질문이 들어오면 (1) **의미검색**(벡터)으로 비슷한 본문을 찾고 → (2) **그래프 탐색**(지분·재무·관계)으로 연결을 따라가고 → (3) **원본 SSOT**(공시 원문·정형 수치)에서 사실을 확정해 → (4) 답변에 **출처(공시 접수번호)** 를 붙여 돌려줍니다.

> SSOT = Single Source of Truth (단일 진실 원천). "같은 사실이 여러 곳에 흩어져 있어도 원본은 한 곳"이라는 규칙.
> GraphRAG = 그래프 + RAG. 벡터검색만 쓰는 일반 RAG와 달리 **관계(그래프)** 까지 따라가서 답변하는 방식.

---

## 1. 3-DB 역할 분담

POLARIS는 **3개의 데이터베이스가 각자 다른 일**을 합니다. 하나만 써서 다 해결하는 대신, 잘하는 일을 나눠 맡깁니다.

| DB | 포트 | 한 줄 역할 | 비유 |
|---|---|---|---|
| **MariaDB** | 3307 | 원본 SSOT + 조인 허브 (사실의 원본) | 도서관의 원본 책장 |
| **Qdrant** | 6333 | 의미검색 (벡터 DB) | "비슷한 내용" 찾아주는 사서 |
| **Neo4j** | 7687 | 관계 그래프 + 멀티홉 탐색 | 인물·기업 관계도 |

### 1.1 왜 3개나 쓰나?

- **관계형(MariaDB)** 만 쓰면: 본문에서 "비슷한 의미"를 못 찾는다. (단어가 안 똑같으면 매칭 실패)
- **벡터(Qdrant)** 만 쓰면: "A의 자회사의 자회사의 매출"같은 **멀티홉**(여러 단계 연결) 질의가 불가능하다.
- **그래프(Neo4j)** 만 쓰면: 원문 텍스트 검색과 정형 수치 조인이 약하다.
→ 세 가지를 합쳐서 각각의 약점을 메운다.

## 2. 교차키 3종 (3개 DB를 묶는 끈)

3개 DB가 서로 다른 정보를 들고 있어도, **공통 식별자** 3개로 같은 사실을 가리킵니다.

| 키 | 형식 | 무엇을 가리키나 | 예시 |
|---|---|---|---|
| `corp_code` | 8자리 숫자 | **기업** 고유코드 (DART가 부여) | `00126380` (삼성전자) |
| `rcept_no` | 14자리 숫자 | **공시 접수번호** = 답변의 출처 근거 | `20250331000123` |
| `chunk_id` | 16자리 hex | **본문 청크** 식별자 (콘텐츠 해시) | `a1b2c3d4e5f60718` |

> "답변에 출처를 붙인다"는 건 결국 이 `rcept_no` 를 함께 돌려준다는 뜻입니다. (DART 뷰어에서 원문 확인 가능)

## 3. 데이터 흐름 (DART → 3-DB)

DART에서 받은 원본 JSON이 어떻게 3개 DB로 퍼지는지의 경로.
핵심은 **원본(`dart_raw_index`)에서 출발해 → 정형(숫자·관계)과 본문(텍스트) 두 갈래로 나뉜다**는 것.

```
DART OpenAPI
   ↓
dart_raw_index   ← 원본 JSON 그대로 보관 (SSOT)
   ├─[정형 데이터]── fin_metric(재무) / 지분 / 임원 ──→ Neo4j (정형 그래프)
   │                                                    [청킹 안 함, 직행]
   │
   └─[본문 데이터]── chunk_index (본문을 청크 단위로 쪼갬)
                       ├─(임베딩: bge-m3)── Qdrant (의미검색)
                       └─(Claude 추출)──── Neo4j (관계 + 근거 청크 링크)
```

포인트:
- **정형 재무·지분·임원**은 이미 숫자가 깔끔하니까 청킹을 거치지 않고 곧장 Neo4j로 들어갑니다.
- **본문**(서술형 텍스트)은 `chunk_index` 에서 작은 조각으로 잘라서 → 임베딩(의미검색용)과 관계추출(Claude) 양쪽으로 갑니다.
- 모든 추출 관계는 **출처 청크(`Chunk`)와 `rcept_no`** 에 연결되어, 나중에 "왜 이렇게 답했는지" 역추적이 가능합니다.

## 4. 질문 → 답변 4단계 (GraphRAG 검색 흐름)

```
[사용자 질문]
   ↓ ① 의미검색 (Qdrant)
        질문을 임베딩하고 polaris-chunks 에서 가까운 청크들 찾기 → chunk_id 목록
   ↓ ② 그래프 탐색 (Neo4j)
        chunk_id 를 기점으로 관계·지분·재무를 멀티홉으로 따라가기
   ↓ ③ 본문/수치 조회 (MariaDB)
        청크 원문(chunk_index)과 정형 수치(fin_metric)를 끌어와 사실 확정
   ↓ ④ 답변 생성
        근거(rcept_no)를 붙여 사용자에게 반환
```

## 6. 핵심 규약 (위반 금지)

1. **데이터 소스 = DART 단일**. 뉴스·SNS·KOSIS 등 다른 소스는 추가하지 않는다.
2. **교차키 형식 일관**: `corp_code(8)`, `rcept_no(14)`, `chunk_id(16hex)`를 모든 DB에서 동일하게.
3. **관계 출처 구분**:
   - `extracted_by IS NULL` → DART 공시에 명시된 **사실**
   - `extracted_by='claude'` → 본문에서 Claude가 **추출한 해석**
4. **근거추적 필수**: 모든 추출 관계는 `chunk_id` → `Chunk` → `rcept_no` 로 원본까지 역추적 가능해야 한다.
5. **숫자·관계는 DB에서**: LLM이 숫자/관계를 새로 만들어내면 안 된다(환각 방어). 항상 DB에서 결정론적으로 꺼낸다.
