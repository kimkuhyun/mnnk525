1. 문제 — 분석가의 진짜 질문은 단일 DB로 답이 안 된다

"한미반도체가 장비/부품을 공급하는 회사는 어디고, 그 근거가 된 공시는 무엇인가?"
"그 공급처(예: SK하이닉스)의 최신 매출 규모는? 한미가 거기에 얼마나 의존하나?"
"한미반도체와 특수관계(소송·분쟁 포함)에 있는 회사는?"

이런 질문은 ① 의미로 문서를 찾고(검색) → ② 회사·인물·재무를 관계로 연결해 따라가고(멀티홉) → ③ 그 답의 출처 공시를 제시(근거)해야 답이 된다. 벡터DB만 쓰면 "비슷한 문서"는 찾지만 "A가 공급하는 회사" 같은 관계를 따라갈 수 없다. 관계형(표)만 쓰면 정형 재무는 되지만 본문에 흩어진 "공급처·특수관계" 같은 추출 관계는 애초에 컬럼이 없고, 의미검색도 못 한다. → POLARIS는 역할을 나눠 MariaDB(사실·원본·근거) + Qdrant(의미검색) + Neo4j(관계 멀티홉) 3-DB로 푼다.

2. 3-DB 역할과 실제 적재 규모

데이터 흐름: DART → dart_raw_index(원본 SSOT) → chunk_index(조인 허브) → Qdrant 임베딩 / Neo4j 관계. 숫자·관계는 DB에서 결정론적으로 가져옴(LLM 생성 금지).

MariaDB 사실·근거

DART 원본 SSOT이자 3-DB 조인 허브. 공시 메타·청크 텍스트·정형 재무·추출 근거 원장.

3. ★ 결정적 — 같은 질문, SQL vs Cypher (실제 결과)

관계형(SQL) — 불가에 가깝다

"공급처"는 공시 본문 서술에서 뽑은 추출 관계다. 관계형 스키마에는 애초에 그런 컬럼/테이블이 없다.

-- 그런 테이블이 없다. 만들려면:
-- (1) 본문을 정규화해 supplier-buyer 표를 직접 설계
-- (2) 근거 문서·신뢰도 컬럼을 매번 수동 관리
-- (3) 회사명 표기 흔들림(에스케이/ SK)을 JOIN 키로 정렬
SELECT ??? -- 원천 데이터 자체가 표가 아님

정형 표만으로는 "본문에서 발견된 관계"를 담을 자리가 없음 → 그래프가 자연스러운 모델.

데모 B — 공급처의 재무까지 한 번에 (지분·재무가 1급 시민)

Cypher — 관계 따라가며 재무 조인

MATCH (h:Organization {corp_code:'00161383'})
      -[s:SUPPLIES_TO]->(b)
      -[:HAS_METRIC]->(m:FinMetric {account_id:'ifrs-full_Revenue',
                                     bsns_year:2025})
RETURN b.name, s.confidence, max(m.value) AS rev2025

비정형 관계(SUPPLIES_TO) → 정형 재무(HAS_METRIC)를 한 패턴으로 횡단. 이게 GraphRAG 멀티홉 핵심.

SQL — 추출관계 표 + 회사매핑 + 재무 다중 JOIN

-- supplies_to 표가 있다고 가정해도:
SELECT b.name, st.confidence, MAX(fm.value)
FROM supplies_to st
JOIN org_resolve r1 ON st.buyer_text = r1.alias
JOIN organization b ON r1.corp_code = b.corp_code
JOIN fin_metric fm ON fm.corp_code = b.corp_code
WHERE st.supplier_code = '00161383'
  AND fm.account_id = 'ifrs-full_Revenue'
  AND fm.bsns_year = 2025
GROUP BY b.name, st.confidence;

표가 늘수록 JOIN 폭증. "공급처의 자회사의 재무" 같은 3홉이면 self-join/CTE가 기하급수로 길어짐. 그래프는 화살표 하나 더면 끝.

벡터DB만 쓰면?

"한미 관련 회사"를 의미검색하면 비슷한 문장이 든 청크는 나오지만, "어느 회사와 어떤 관계인지"를 정확히 열거·집계할 수 없다.

그래프는 "관계 = 1급 데이터"라서 정확히 4건, confidence, 근거 공시까지 결정론적으로 반환. 환각 여지 없음.

5. 정직한 경계 — 할 수 있는 것 / 못 하는 것

할 수 있다 (공시에 명시된 것)

  • 지분 관계 (IS_MAJOR_SHAREHOLDER_OF, qota_rt)
  • 특수관계자 (RELATED_PARTY — 계열·소송 당사자 포함)
  • 계열·자회사 (IS_SUBSIDIARY_OF), 출자 (INVESTS_IN)
  • 임원 (EXECUTIVE_OF), 재무지표 (HAS_METRIC, IFRS)
  • 제품·기술 (PRODUCES, USES_TECH), 공급 관계 (SUPPLIES_TO)
  • 모든 추출 관계에 근거 chunk_id·rcept_no·confidence 동봉

못 한다 (공시 범위 밖)

  • 차명·실소유주(UBO) 추적 — 공시에 안 나옴
  • 소액주주 전체 명부 — 주요주주만 공시
  • 거래 금액의 건별 정밀 매칭 — 본문은 정성 서술 위주
  • 비공개 계약·미공시 거래
  • 실시간 시세·뉴스 — 데이터 소스는 DART 공시 단일

경계를 명확히 하는 이유: 답마다 출처를 붙이고, 모르는 건 모른다고 하기 위함. 숫자·관계는 LLM이 생성하지 않고 DB가 결정한다.
