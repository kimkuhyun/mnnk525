## 총평 (팀장)

코어는 탄탄하다 — 3-DB GraphRAG가 실제로 돌고, 근거(evidence) 드릴다운·출처구분(공시 vs 언급)·Blue/Green 골격·Contextual Retrieval 같은 설계 의도는 수준급이다. 다만 "데모는 되는데 배포·확장에서 깨지는" 찐빠가 여럿 보인다. 크게 다섯 갈래:

1. **5사인데 3사만 산다.** 동진쎄미켐·솔브레인이 코드 곳곳의 하드코딩 3사 맵 때문에 누락된다 — 프론트 회사선택(`CompanyContext`), 뉴스 매칭(`news_crawl/load` CORP_MAP), 감성·다이제스트(NAMES). **데이터는 적재됐는데 화면·분석에서 빠진다.** → `config.CORPS`/`corps.json` 단일 소스로 통일.
2. **document_unified 통합레이어 우회.** chunk 로더 5종(news/ftc/kosis/bok/krx) + `ir_report` 가 통합 레이어를 건너뛰고 직접 적재 — CLAUDE.md 2번 규약의 광범위 위반. 게다가 뉴스 적재 경로가 둘(`chunk/news.py` 구경로 vs `ingest/news_crawl` v2)로 이원화돼 서로 다른 컬렉션·테이블에 쓴다.
3. **타환경/배포 미고려.** 프론트가 Tailwind·Pretendard를 런타임 CDN으로 의존(빌드 시 스타일 증발), backend 컨테이너에 pola가 없어 스케줄러 잡 전멸, docker-compose가 DB 자격증명 미주입, 진단스크립트에 `C:\Users\kimkuhyn` 절대경로, dev 비밀번호·RUN_ID·green 컬렉션 하드코딩.
4. **스키마 drift.** init SQL에 DDL을 모았지만 모듈 `ensure_tables`가 9곳 잔존(일부 collation/타입 drift), `ir_report` DDL이 3곳에서 상이, Neo4j `Company` 이중라벨이 SOT 문서에 누락.
5. **안전망 부재.** 파이프라인·BE·FE 테스트 사실상 0, DB 커넥션 `try/finally` 누락 다발(예외 시 누수).

## 서브시스템별 잘 된 점

- **데이터 수집 (ingest)**: 뉴스 v2 파이프라인(news_crawl)이 doc_id = sha1("news:"+news_id)[:16] 단일 정합키로 MariaDB document_unified → Qdrant(point.id=int(doc_id,16)) → Neo4j(:Document)-ABOUT->(:Organization) → mention_daily 를 일관되게 연결한 설계가 깔끔하다. CLAUDE.md 의 통합레이어 규약과 출처구분(graph_load EXTRACTED_BY='claude', rel_query 의 doc_ids 누적 멱등)을 정확히 지킨다. RSS undefined-entity sanitize(_sanitize_rss: named entity 사전 + bare & 정규식)와 한경/매경 사이트별 우회는 메모리에 기록된 실전 노하우가 코드에 잘 반영됐다. DART 수집(bulk_collect)의 on-disk 캐시 + tenacity 지수백오프 + status 코드 분류(DART_STATUS_MAP) + rate-profile(slow/normal/fast) + 멱등 upsert(ON DUPLICATE KEY) 도 견고하다.
- **청킹·임베딩 (chunk/embed)**: Contextual Retrieval 구현이 정석이다 — text.py:79-80·table.py:98-99 에서 embedding_text 에 "회사명+문서종류+섹션헤더" prefix 를 붙이고 chunk_id 등 식별자는 절대 임베딩에 넣지 않으며(설계 05 §5 준수), llm_context_text/payload 와 3필드를 깔끔히 분리한다. 토큰 카운트가 임베딩 모델과 정확히 일치하는 bge-m3 토크나이저 기반(semantic_split.py)이라 8192 한도 대비 2800 안전마진 설정이 신뢰도 높다. bge_m3.py 의 임베딩 검증(norm/​NaN/​shape, /api/show digest 기록으로 :latest 재현성 확보)과 embed_batch 의 size-mismatch 가드·재시도는 견고하다. chunk_id 가 content_sha1 을 포함한 immutable 설계라 재생성 멱등성이 보장된다.
- **그래프 추출 (graph)**: LLM 구조화 출력 방어가 견고하다: entity/relation 모두 format=<JSON Schema> + think=False + num_ctx 확장 + temperature 0 을 일관 적용하고(사용자 Ollama 노하우와 정합), 6단 검증(substring 위조 검증·type enum·confidence 임계·중복키)으로 환각을 강하게 차단한다. PROV 재현성 설계가 인상적이다 — manifest 에 git_commit·prompt_hash(SHA1)·lexicon SHA1·실제 모델 digest(/api/show 의 modelfile FROM 파싱 fallback 포함)를 스냅샷하고, ExtractionActivity 노드로 wasGeneratedBy 를 건다. ADR-008 의 :LLMExtracted 라벨 격리(롤백 1줄, 정형 F1 게이트 보호)와 ER 링킹의 단계 카운터(_LINK_STATS) + Stage2b loose 재시도(canonical exact-match 면 임계 무시) 설계도 실전 함정을 반영한 좋은 엔지니어링이다. qdrant .search()→.query_points() 회귀를 ADR 주석으로 남긴 점도 모범적.
- **DB 적재·운영 (db)**: Blue/Green standby→active 모델이 일관되게 관철됨: run_id 단일 진실(active_run_manifest)에서 발급하고 세 DB(MariaDB chunk_index, Qdrant payload, Neo4j run-scoped 노드)가 같은 run_id 를 공유하며, 모든 적재가 UPSERT/MERGE 라 멱등하다. ingest_status='pending'→'ready' 2단계 게이트로 부분 적재가 검색에 노출되지 않게 막은 설계가 견고하다. graph_diag.py 의 C-01~C-15 진단은 PROV 연결·고아 노드·run_id 일관성·LLMExtracted 격리까지 커버하는 수준 높은 read-only 무결성 스위트이고, init_neo4j.py 의 entity(전역) vs run-scoped 제약 분리도 설계 문서와 잘 맞는다. 예외 시 rollback 하는 init_mariadb.py, FOR UPDATE 로 standby 슬롯을 잠그는 acquire_standby_run_id 등 트랜잭션 감각도 곳곳에 보인다.

- **백엔드 라우터 (backend/routers)**: SQL/Cypher 파라미터화가 전반적으로 모범적이다 — IN 절은 항상 `",".join(["%s"]*n)` 플레이스홀더 + 튜플 바인딩, Cypher 는 `$param` 으로만 값 주입해 인젝션 표면이 사실상 없다(graph/node/evidence/changes 전부). response_model 이 모든 엔드포인트에 명시돼 /docs 스키마·프론트 types.ts 와 미러가 유지되고, relations.py 의 PREDICATE_TO_GROUP/DIRECTED 를 단일 출처로 공유한다.
